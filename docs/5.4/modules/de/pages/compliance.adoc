== Management von Compliance und CIS-Benchmarks

Die Compliance-Prüfung mit {product-name} umfasst CIS-Benchmarks, benutzerdefinierte Prüfungen, die Prüfung von Geheimnissen und branchenübliche Vorlagen für PCI, GDPR und andere Vorschriften.

Die CIS-Benchmarks werden automatisch von {product-name} ausgeführt:

* Kubernetes
* Docker
* Red Hat OpenShift-Entwurf "Inspired by CIS"-Benchmarks
* Google GKE

Die Ergebnisse der Konformitätsprüfung können für einzelne Assets in den Menüs Register (für Bilder), Knoten und Container angezeigt werden, indem Sie das entsprechende Asset auswählen und auf die Registerkarte Konformität klicken.

Das Menü Sicherheitsrisiken -> Compliance ermöglicht konsolidierte Compliance-Berichte, ähnlich wie das Menü Schwachstellen funktioniert.

=== Sicherheitsrisiken - Compliance und Compliance-Profil

Die Übereinstimmungsergebnisse werden in der Liste nach Kategorie und Name angezeigt. Zu den Kategorien gehören Docker, Kubernetes, OpenShift und Custom. Die Namen der einzelnen Punkte entsprechen den CIS-Benchmarks. So entspricht beispielsweise K.4.2.3 dem Kubernetes-CIS-Benchmark 4.2.3. Den Docker-Benchmarks wird ein "D" vorangestellt, mit Ausnahme der bildbezogenen Benchmarks, denen ein "I" vorangestellt wird.

Verwenden Sie den erweiterten Filter, um Konformitätsprüfungen auf der Grundlage von Plattform, Host, Namespace oder Industriestandard auszuwählen, wie unten dargestellt.

image:compliance_4.png[compliance]

Nach Anwendung des Filters werden nur die relevanten CIS-Benchmarks und benutzerdefinierten Prüfungen angezeigt, und es kann ein Bericht erstellt und heruntergeladen werden. Auf diese Weise können Berichte für Standards wie PCI, HIPAA, GDPR und andere Standards erstellt werden.

=== Anpassung von Compliance-Vorlagen für PCI, GDPR, HIPAA, NIST, PCIv4 und DISA STIG

Das Menü Compliance-Profil ermöglicht die Anpassung der integrierten Vorlagen für Branchenstandards wie PCI, GDPR, HIPAA, NIST, PCIv4 und DISA STIG. Diese Berichte können über das Menü "Sicherheitsrisiken" -> "Compliance" erstellt werden, indem Sie einen der zu filternden Standards auswählen und dann exportieren. Das NIST-Profil ist für https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-190.pdf[SP 800-190] NIST.

Um ein beliebiges Konformitätsprofil anzupassen, wählen Sie den Industriestandard (z. B. PCI) und aktivieren oder deaktivieren Sie dann bestimmte Prüfungen für diesen Standard. Stellen Sie sich diese als Konformitätskennzeichen vor, die auf jede Prüfung angewendet werden, um einen Konformitätsbericht für diesen Industriestandard zu erstellen.

Verwenden Sie die Schaltfläche Aktion, um eine Kennzeichnung für die Einhaltung der Vorschriften zu dieser Prüfung hinzuzufügen oder zu entfernen.

image:compliance_profile_4.png[Profil]

Darüber hinaus können Sie auf der Registerkarte "Assets" auswählen, welche "Assets" in die Compliance-Berichte aufgenommen werden sollen. Standardmäßig werden alle Compliance-Vorlagen auf Bilder, Knoten und Container angewendet.

image:profile_assets_4.png[compliance_assets]

Verwenden Sie die Schaltfläche Aktion, um Compliance-Vorlagen für Assets hinzuzufügen oder zu entfernen.

* Bilder. Wählen Sie den/die Standard(s) aus, der/die für Bilder gemeldet werden soll(en).
* Knotenpunkte. Wählen Sie den/die Standard(s), der/die für Knoten (Hosts) gemeldet werden soll(en).
* Behältnisse. Wählen Sie den/die Standard(s), der/die für Container gemeldet werden soll(en).

Alternativ zu den oben genannten Kriterien können die Konformitätsvorlagen auch auf bestimmte Namensräume beschränkt werden. Wenn dieses Kästchen markiert ist und ein oder mehrere Namensräume hinzugefügt wurden, werden Berichte für alle Assets erstellt, die für diese Namensräume gelten. Dies kann nützlich sein, wenn das PCI-Template beispielsweise nur über Assets für Namensräume berichten soll, die PCI-in-scope (anwendbare) Workloads enthalten.

image:DocheckASSETSEnableNS.png[Namensraum]

Nachdem die Vorlagen und Assets (falls gewünscht) im Menü Sicherheitsrisiken -> Compliance-Profile angepasst wurden, können im Menü Sicherheitsrisiken -> Compliance Berichte erstellt werden, indem der erweiterte Filter geöffnet und die gewünschte Compliance-Vorlage ausgewählt wird. Wenn Sie zum Beispiel GDPR auswählen, werden die Anzeige und die Berichte nur für das GDPR-Profil gefiltert.

=== Beispielhafte Helm-Installation mit Prime Compliance

Im Folgenden finden Sie ein Beispiel für einen Helm-Installationsbefehl zur Installation von SUSE Security 5.4.0 mit Prime-Compliance. Die Benutzer können die Installationsversion auf 5.4.0 oder höher ändern, da die Prime Compliance mit 5.4.0 beginnt. Nach der Installation können Prime-Benutzer die verfügbaren Vorschriften in der Web-UI von SUSE Security auf der Registerkarte **Compliance > Vorschriften** überprüfen.

[,shell]
----
helm install neuvector -n neuvector ./ --set controller.prime.enabled=true --set
controller.prime.image.repository=nvlab/compliance_config,controller.prime.image.tag=1.
0.0 --set
tag=5.4.0-b2,controller.image.repository=nvpublic/co,enforcer.image.repository=nvpublic/
en,manager.image.repository=nvpublic/ma
----

=== Geheimnisse Auditing

{product-name} prüft im Rahmen von Image-Compliance-Scans und Laufzeit-Scans auf über 40 gängige Arten von Geheimnissen. Darüber hinaus können xref:customcompliance.adoc[benutzerdefinierte Compliance-Skripte] für Container oder Hosts konfiguriert werden, und die xref:dlp.adoc[DLP-Paketinspektionsfunktion] kann verwendet werden, um nach Geheimnissen in Netzwerk-Nutzdaten zu suchen.

Die Ergebnisse der Prüfung von Geheimnissen finden Sie im Abschnitt Compliance von Image-Scans (Assets -> Registries), Containern (Assets -> Containers), Knoten (Assets -> Nodes) und im Menü Compliance Management (Security Risks -> Compliance).

Nachfolgend finden Sie ein Beispiel dafür, wie bei einem Bildscan entdeckte Geheimnisse angezeigt werden.

image:secrets_image_4.png[Geheimnisse]
